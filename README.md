# London Live Bus Journey Tracker

A real-time London bus tracking Android app built with Jetpack Compose, Clean Architecture, and the TfL Unified API.

## Features

- 🔍 **Journey Planning** - Search locations and plan bus journeys with disambiguation handling
- 🚌 **Live Bus List** - View real-time bus arrivals for any route
- 📍 **Virtual GPS Tracking** - Track bus position in real-time using inferred coordinates
- 🗺️ **Google Maps Integration** - Interactive maps with custom markers and route polylines
- 🎨 **Material 3 Design** - Modern, accessible UI following Figma specifications

## Setup Instructions

### Prerequisites

- Android Studio Panda 
- Android SDK
- Kotlin

### 1. Clone the Repository

```bash
git clone https://github.com/THAMMY202/LondonLiveBusJourneyTracker.git
cd LondonLiveBusJourneyTracker
```

### 2. Configure API Keys

The app requires two API keys stored in `local.properties`:

```bash
# Copy the template file
cp local.properties.template local.properties
```

Edit `local.properties` in the project root:

```properties
# Android SDK location (auto-generated by Android Studio)
sdk.dir=/path/to/your/Android/Sdk

# ===========================================
# TfL API Key (REQUIRED)
# ===========================================
# Register at: https://api-portal.tfl.gov.uk/
TFL_API_KEY=your_tfl_api_key_here

# ===========================================
# Google Maps API Key (REQUIRED)
# ===========================================
# Get from: https://console.cloud.google.com/
MAPS_API_KEY=your_google_maps_api_key_here
```

> ⚠️ **Important**: `local.properties` is gitignored to protect the API keys. Never commit this file.

### 3. Get TfL API Key

1. Go to [TfL API Portal](https://api-portal.tfl.gov.uk/)
2. Click **Sign Up** and create an account
3. Once logged in, go to **Products** → **500 Requests per min**
4. Click **Subscribe** (it's free)
5. Go to **Profile** → **Subscriptions**
6. Copy your **Primary Key**
7. Paste into `local.properties` as `TFL_API_KEY`

### 4. Get Google Maps API Key

1. Go to [Google Cloud Console](https://console.cloud.google.com/)
2. Create a new project or select existing
3. Navigate to **APIs & Services** → **Library**
4. Search and enable **Maps SDK for Android**
5. Go to **APIs & Services** → **Credentials**
6. Click **Create Credentials** → **API Key**
7. Click on the created key to configure restrictions:
    - **Application restrictions**: Android apps
    - **Add an item**:
        - Package name: `com.tracker.londonbusjourney`
        - SHA-1 certificate fingerprint (get via `./gradlew signingReport`)
    - **API restrictions**: Restrict to Maps SDK for Android
8. Copy the key and paste into `local.properties` as `MAPS_API_KEY`

### 5. Build and Run

**Via Android Studio:**
1. Open the project in Android Studio
2. Sync Gradle files
3. Select a device/emulator
4. Click Run ▶️

**Via Command Line:**
```bash
./gradlew assembleDebug
adb install app/build/outputs/apk/debug/app-debug.apk
```

## Lead-Level Design Questions

### 1. Vehicle Positioning Logic

**How did you infer vehicle position from the available data?**

The TfL API does not expose real-time GPS coordinates for buses. I implemented a **"Virtual GPS" algorithm** that infers position from arrival predictions:

```
Algorithm:
1. Cache route sequence on init (GET /Line/{id}/Route/Sequence/{direction})
   → Returns ordered list of stops with lat/lon coordinates
   
2. Poll arrivals every 30 seconds (GET /Line/{id}/Arrivals)
   → Returns predictions with vehicleId, naptanId (stop), timeToStation
   
3. Match vehicleId to its next stop (naptanId)
   
4. Look up stop coordinates from cached route sequence
   
5. Use stop's lat/lon as inferred bus position
```

**Implementation:**
```kotlin
// BusRepositoryImpl.kt
override suspend fun getBusPosition(vehicleId: String, lineId: String, routeSequence: RouteSequence): Result<BusPosition> {
    val arrivals = getArrivalsForLine(lineId)
    val vehicleArrival = arrivals.find { it.vehicleId == vehicleId }
    val currentStop = routeSequence.findStopById(vehicleArrival.naptanId)
    
    return BusPosition(
        lat = currentStop.lat,
        lon = currentStop.lon,
        currentStopName = currentStop.name,
        timeToNextStopSeconds = vehicleArrival.timeToStation
    )
}
```

**Assumptions & Limitations:**

| Assumption | Limitation |
|------------|------------|
| Bus is at or near its next predicted stop | Position "jumps" between stops rather than smooth interpolation |
| Route sequence is static | Diversions or service changes not handled |
| Vehicle continues on same route | No handling if bus goes out of service |
| 30s polling is sufficient | Position can be up to 30s stale |

**Potential Improvements:**
- Interpolate position between stops using `timeToStation`
- Use previous position + time delta for smoother animation
- Fallback to route polyline midpoint between stops

### 2. Journey Disambiguation Handling

**How did you model and resolve ambiguous locations or journey results?**

The TfL API returns disambiguation when a location query matches multiple places (e.g., "Victoria" matches Victoria Station, Victoria Coach Station, Victoria Street, etc.).

**Detection:**
```kotlin
// JourneyPlannerResponse.kt
data class JourneyPlannerResponse(
    val journeys: List<JourneyDto>? = null,
    val fromLocationDisambiguation: DisambiguationDto? = null,
    val toLocationDisambiguation: DisambiguationDto? = null
)
```

**Resolution Strategy:**

1. **Location Search Phase** (`SearchLocationsUseCase`):
    - Use `GET /StopPoint/Search/{query}?modes=bus` for initial search
    - Return list of matching locations with ICS codes
    - UI presents options if multiple matches exist

2. **Journey Planning Phase** (`GetJourneyOptionsUseCase`):
    - Use selected location's `icsId` (ICS code) for unambiguous journey request
    - If API still returns disambiguation, map options to domain model

3. **Fallback Handling**:
    - If `icsId` unavailable, use coordinates: `"{lat},{lon}"`
    - If coordinates unavailable, use original NaPTAN ID

**Data Flow:**
```
User types "Victoria"
    ↓
SearchLocationsUseCase → Multiple StopPointMatchDto results
    ↓
UI shows: "Victoria Station", "Victoria Coach Station", etc.
    ↓
User selects "Victoria Station" (icsId: "1000248")
    ↓
Journey request uses icsId → Unambiguous result
```

**UI State Model:**
```kotlin
data class JourneySearchUiState(
    val fromSuggestions: List<Location> = emptyList(),
    val toSuggestions: List<Location> = emptyList(),
    val selectedFrom: Location? = null,  // Contains resolved icsId
    val selectedTo: Location? = null
)
```

### 3. Architecture Decisions

**Why MVVM + Clean Architecture?**

| Decision | Rationale |
|----------|-----------|
| **MVVM** | Native fit with Compose's reactive model; `StateFlow` maps directly to Compose state |
| **Clean Architecture** | Separates concerns; domain layer is framework-agnostic and testable |
| **Use Cases** | Single-responsibility; orchestrate business logic; easy to test in isolation |
| **Repository Pattern** | Abstracts data source; enables caching and offline support |
| **Hilt DI** | Compile-time safety; native Android lifecycle integration |

**Polling Coordination with UI State:**

```kotlin
// BusListViewModel.kt
class BusListViewModel @Inject constructor() : ViewModel() {
    
    private var pollingJob: Job? = null
    private val _uiState = MutableStateFlow(BusListUiState())
    val uiState: StateFlow<BusListUiState> = _uiState.asStateFlow()
    
    private fun startPolling() {
        pollingJob?.cancel()
        pollingJob = viewModelScope.launch {
            while (isActive) {
                loadBusArrivals()  // Updates _uiState
                delay(30_000L)     // 30 second interval
            }
        }
    }
    
    fun resumePolling() {
        if (pollingJob?.isActive != true) startPolling()
    }
    
    fun stopPolling() {
        pollingJob?.cancel()
    }
    
    override fun onCleared() {
        stopPolling()
    }
}

// BusListScreen.kt - Lifecycle awareness
DisposableEffect(lifecycleOwner) {
    val observer = LifecycleEventObserver { _, event ->
        when (event) {
            Lifecycle.Event.ON_RESUME -> viewModel.resumePolling()
            Lifecycle.Event.ON_PAUSE -> viewModel.stopPolling()
            else -> { }
        }
    }
    lifecycleOwner.lifecycle.addObserver(observer)
    onDispose { lifecycleOwner.lifecycle.removeObserver(observer) }
}
```

**Key Patterns:**
- `viewModelScope` ensures coroutines are cancelled when ViewModel is cleared
- `isActive` check prevents emission after cancellation
- Lifecycle observer pauses polling when app is backgrounded (saves battery/data)
- `StateFlow` ensures UI always has latest state

### 5. Scaling Consideration

**How might this solution change if vehicle-positioning logic were moved server-side?**

**Current Client-Side Approach:**
```
Client polls /Arrivals → Client matches to route → Client infers position
```

**Server-Side Approach:**
```
Client subscribes to /VehiclePositions/{vehicleId}
Server pushes: { lat, lon, heading, speed, nextStop, eta }
```

**Architectural Changes:**

| Aspect | Current (Client) | Server-Side |
|--------|-----------------|-------------|
| **Data Fetching** | Polling every 30s | WebSocket or SSE push |
| **Position Logic** | `BusRepositoryImpl.getBusPosition()` | Removed; use server response |
| **Route Sequence** | Cached client-side | Optional; server provides coordinates |
| **Battery/Data** | Higher (constant polling) | Lower (event-driven) |
| **Accuracy** | Stop-based approximation | True GPS coordinates |
| **Offline** | Can show last known position | Requires connection |

**Code Changes:**

```kotlin
// Current
class TrackBusPositionUseCase {
    operator fun invoke(vehicleId: String, lineId: String): Flow<Result<BusPosition>> = flow {
        val route = repository.getRouteSequence(lineId)
        while (true) {
            emit(repository.getBusPosition(vehicleId, lineId, route))
            delay(30_000)
        }
    }
}

// Server-side positioning
class TrackBusPositionUseCase {
    operator fun invoke(vehicleId: String): Flow<Result<BusPosition>> {
        return repository.subscribeToVehiclePosition(vehicleId)  // WebSocket stream
    }
}

// Repository
interface BusRepository {
    fun subscribeToVehiclePosition(vehicleId: String): Flow<Result<BusPosition>>
}

class BusRepositoryImpl {
    override fun subscribeToVehiclePosition(vehicleId: String): Flow<Result<BusPosition>> {
        return webSocketClient.connect("/vehicles/$vehicleId/position")
            .map { frame -> mapper.mapToBusPosition(frame) }
    }
}
```

**Benefits of Server-Side:**
- Real GPS accuracy (not stop-based approximation)
- Smooth position interpolation possible
- Reduced client complexity
- Centralized logic updates without app releases
- Better support for features like ETA prediction, crowding data

**Trade-offs:**
- Requires backend infrastructure investment
- WebSocket connection management complexity
- Potential latency vs polling (depends on implementation)
- Server costs scale with concurrent users
